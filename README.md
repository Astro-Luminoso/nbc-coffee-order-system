# NBC Coffee Order System

포인트로 커피를 주문·결제하고, 주문 데이터를 수집 플랫폼으로 전달하며, 최근 인기 메뉴를
조회하는 Spring Boot 기반 백엔드입니다.

이 문서는 과제의 필수 요구사항과 도전 요구사항을 해결하기 위해 선택한 설계와 구현 전략을
설명합니다. 상세 API 계약은 [API 명세서](docs/ko/API.md), 관계형 모델은
[ERD](docs/ko/ERD.md), 계층과 처리 흐름은 [아키텍처 문서](docs/ko/ARCHITECTURE.md)를
참고합니다.

## 구현 기능

| 기능 | 메서드와 경로 | 설명 |
|---|---|---|
| 커피 메뉴 조회 | `GET /api/v1/menus` | 메뉴 ID, 이름, 가격을 ID 오름차순으로 조회합니다. |
| 포인트 충전 | `POST /api/v1/users/{userId}/point-charges` | 사용자 포인트를 충전합니다. `1원 = 1P`이며 멱등키가 필요합니다. |
| 주문·결제 | `POST /api/v1/orders` | 포인트를 차감하고 단일 메뉴 주문을 생성합니다. 멱등키가 필요합니다. |
| 인기 메뉴 조회 | `GET /api/v1/menus/popular` | 서울 시간 기준 직전 완료 7일의 메뉴별 주문 수를 합산해 상위 3개를 반환합니다. |

모든 API는 공통 응답 형식을 사용하며, 요청 검증·오류 코드·멱등성 헤더 규칙은
[컨벤션](docs/ko/CONVENTION.md)에 정의돼 있습니다.

## 설계 내용

### 도메인과 저장소

MySQL은 사용자 포인트, 메뉴, 주문, 멱등성 기록의 기준 데이터 저장소입니다. Redis는 인기
메뉴 조회 성능을 위한 재구축 가능한 캐시입니다.

```text
User ──< CoffeeOrder >── Menu
                │
                └── IdempotencyRecord

MySQL: 사용자·메뉴·주문·멱등성 기록의 기준 데이터
Redis: 날짜별 인기 메뉴 ZSET 캐시
```

주문은 현재 API의 입력이 단일 `menuId`이므로 주문 항목 테이블을 별도로 만들지 않았습니다.
주문에는 결제 당시의 가격(`paymentAmount`)을 저장하여 메뉴 가격이 나중에 변경돼도 과거
결제 이력이 변하지 않게 합니다.

### 주문과 데이터 수집 흐름

```text
주문 결제 트랜잭션
  ├─ 포인트 조건부 차감
  ├─ 주문 생성 (수집 상태: PENDING)
  └─ 멱등성 결과 저장

트랜잭션 커밋
  └─ 수집 플랫폼 전송 즉시 시도
       ├─ PENDING 주문 행을 DB 비관적 잠금으로 조회
       ├─ 성공: 수집 상태를 SUCCEEDED로 변경
       └─ 실패: PENDING 유지 → 스케줄러가 재시도
```

수집 데이터에는 `orderId`, `userId`, `menuId`, `paymentAmount`를 포함합니다.
기본 구현은 로그 기반 수집 클라이언트이며, 통합 테스트에서는 mock client로 실제 전달 시점과
실패 후 재시도를 검증합니다. 전송 서비스는 아직 `PENDING`인 주문만 DB 비관적 잠금으로
가져오므로 여러 인스턴스가 정상 경합으로 같은 주문을 동시에 전송하지 않습니다. 다만 외부
호출 성공과 `SUCCEEDED` 커밋은 하나의 원자 작업이 아니므로 전달은 at-least-once 방식이며,
수집 플랫폼은 `orderId`를 기준으로 중복 수신을 무시해야 합니다.

## 설계 의도와 문제 해결 전략

### 1. 포인트와 주문의 동시성

포인트는 애플리케이션 메모리에서 읽고 수정한 뒤 저장하지 않습니다. 대신 DB에서 원자적
증가 또는 조건부 감소 쿼리를 실행합니다.

- 충전: `point_balance = point_balance + amount`
- 결제: `point_balance >= price`일 때만 `point_balance = point_balance - price`

따라서 여러 요청이 동시에 도착해도 충전 갱신이 유실되지 않고, 동일 사용자가 동시에 주문해도
잔액보다 많은 포인트를 사용할 수 없습니다.

### 2. 멱등성으로 재시도 안전성 확보

포인트 충전과 주문 결제는 `Idempotency-Key` 헤더를 요구합니다. 작업 종류와 키의 복합
유니크 제약으로 여러 인스턴스의 중복 요청을 직렬화하고, 요청 본문의 SHA-256 해시로 같은
키가 다른 요청에 재사용되는 것을 차단합니다.

같은 요청의 재시도는 최초 성공 응답을 반환하므로 포인트가 두 번 충전되거나 주문이 중복으로
생성되지 않습니다.

### 3. 결제와 주문 생성의 일관성

주문 결제는 하나의 DB 트랜잭션에서 다음을 수행합니다.

1. 멱등성 기록을 잠금 조회합니다.
2. 메뉴 가격을 확인하고 포인트를 조건부 차감합니다.
3. 주문과 결제 시점의 금액을 저장합니다.
4. 최초 성공 응답을 멱등성 기록에 저장합니다.

잔액 부족이나 예외가 발생하면 트랜잭션 전체가 롤백되므로 포인트만 차감되거나 주문만
생성되는 상태를 막습니다. 외부 수집 전송은 커밋 후에만 시도하므로 실패한 결제의 주문 정보가
전송되지 않습니다.

### 4. 인기 메뉴 정확성과 복구

인기 메뉴는 Redis의 날짜별 ZSET(`popular-menu:{yyyy-MM-dd}`)으로 관리합니다. 주문이
커밋된 후 Redis Lua 스크립트가 주문별 projection marker를 생성하고 점수를 증가시켜, 재시도
이벤트가 같은 주문을 두 번 집계하지 못하게 합니다.

Redis 키가 없거나 형식이 올바르지 않으면 MySQL의 커밋된 주문을 날짜·메뉴별로 다시 집계해
캐시를 재구축합니다. 재구축과 projection은 날짜별 Redis 분산 락 안에서 실행해 여러
인스턴스가 동시에 갱신해도 카운트가 어긋나지 않도록 했습니다.

### 5. 다중 인스턴스 환경

정확성 판단을 단일 JVM 메모리에 두지 않습니다.

- MySQL의 원자 쿼리, 트랜잭션, 유니크 제약, 비관적 잠금으로 포인트·멱등성·수집 전송을 조정합니다.
- Redis의 Lua 스크립트와 날짜별 분산 락으로 인기 메뉴 projection을 조정합니다.
- 수집 전송은 잠긴 `PENDING` 주문의 영속 상태를 기준으로 재시도합니다. 외부 호출 성공 뒤
  DB 커밋 실패로 가능한 중복만 수신 측의 `orderId` 중복 제거로 복구합니다.

## 기술 선택 이유

| 기술 | 선택 이유 |
|---|---|
| Spring Boot / Spring MVC | 검증, 예외 처리, 트랜잭션, HTTP API를 일관된 방식으로 구성하기 위해 사용했습니다. |
| MySQL / JPA | 포인트·주문·멱등성처럼 강한 일관성이 필요한 데이터를 트랜잭션으로 관리하기 위해 사용했습니다. |
| Redis ZSET | 날짜별 메뉴 주문 횟수를 점수로 누적하고 상위 메뉴를 빠르게 조회하기 위해 사용했습니다. |
| Redis Lua / 분산 락 | 여러 인스턴스의 projection 중복 증가와 캐시 재구축 경합을 제어하기 위해 사용했습니다. |
| 멱등성 기록 | 네트워크 재시도나 중복 요청이 포인트·주문에 중복 부작용을 만들지 않게 하기 위해 사용했습니다. |
| 커밋 후 이벤트, DB 행 잠금, 재시도 스케줄러 | 외부 수집 전송 실패가 결제를 되돌리지 않으면서도 이후 재시도하고, 정상적인 인스턴스 경합의 중복 호출을 막기 위해 사용했습니다. |

## 검증 범위

`./gradlew test`로 단위·통합 테스트를 실행합니다. 주요 검증 항목은 다음과 같습니다.

- 메뉴 조회 응답과 정렬
- 포인트 충전, 요청 검증, 사용자 없음, 멱등키 재사용
- 동시 충전에서의 갱신 유실 방지
- 주문 결제, 잔액 부족 시 롤백, 주문 멱등성
- 커밋 후 수집 전송, 전송 실패 시 `PENDING` 유지, 스케줄러 재시도, 동일 주문 동시 전송 1회 보장
- 최근 7일 인기 메뉴 집계, 동점 정렬, Redis 캐시 누락 및 projection 실패 복구

## 로컬 실행

Java 21, Gradle, Docker가 필요합니다. Redis 실행과 로컬 프로필 설정은
[로컬 개발 가이드](docs/ko/local-dev.md)를 참고합니다.

```bash
./gradlew bootRun --args='--spring.profiles.active=local'
```
