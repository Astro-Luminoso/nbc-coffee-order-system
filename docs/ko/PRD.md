# 커피 주문 시스템 제품 요구사항 문서

> 원문: [English PRD](../PRD.md)

## 1. 프로젝트 목표

이 과제의 목표는 신뢰할 수 있는 커피 주문 서비스 백엔드를 만드는 것입니다. 사용자는 커피
메뉴를 둘러보고, 포인트를 충전하고, 포인트로 커피를 주문·결제하고, 현재 날짜 직전의 완료된
7개 달력 날짜에서 가장 인기 있는 메뉴 3개를 조회할 수 있어야 합니다.

해결책은 기본 API 동작만 보여서는 안 됩니다. 여러 애플리케이션 인스턴스가 실행돼도 정확해야
하고, 동시성 문제로부터 포인트 잔액과 주문 수를 보호해야 하며, 데이터 일관성을 보존하고,
모든 기능과 제약을 자동화 테스트로 다뤄야 합니다.

### 성공 기준

- 필수 API 4개가 이 문서에 따라 구현되고 동작합니다.
- 1원은 1포인트로 변환됩니다.
- 커피 구매는 포인트로만 결제할 수 있습니다.
- 주문 생성과 결제 금액 차감은 일관되게 처리됩니다.
- 포인트 충전과 결제는 멱등키로 안전하게 재시도할 수 있습니다.
- 성공적으로 결제된 모든 주문은 결제 트랜잭션에서 대기 전송 상태로 기록되고, 커밋 뒤
  MockAPI.io의 `orders` 리소스와 조정됩니다. 전송 실패는 결제를 롤백하지 않고 재시도됩니다.
- 인기 메뉴 상위 3개는 `Asia/Seoul` 업무 시간대 기준 현재 날짜 직전 완료 7일의 정확한 결제
  주문 수로 계산하며, 현재 날짜는 제외합니다.
- 여러 서버 인스턴스에서 애플리케이션이 올바르게 동작합니다.
- 자동화 테스트가 기능, 업무 규칙, 동시성, 일관성 제약을 다룹니다.

## 2. 사용자 시나리오

### 시나리오 1: 커피 메뉴 둘러보기

1. 사용자가 커피 메뉴 목록을 요청합니다.
2. 시스템은 각 메뉴의 ID, 이름, 가격을 반환합니다.
3. 사용자는 주문할 메뉴를 선택합니다.

### 시나리오 2: 포인트 충전

1. 사용자가 사용자 식별자, 양수 충전 금액, 멱등키를 제출합니다.
2. 시스템은 `1원 = 1포인트`로 변환해 사용자 잔액에 더합니다.
3. 시스템은 갱신된 포인트 잔액을 반환합니다.
4. 같은 키와 같은 요청을 반복하면 포인트를 다시 더하지 않고 최초 결과를 반환합니다.
5. 동시 충전이 발생해도 포인트 갱신이 유실되지 않아야 합니다.

### 시나리오 3: 커피 주문·결제

1. 사용자가 사용자 식별자, 메뉴 ID, 멱등키를 제출합니다.
2. 시스템은 메뉴와 현재 가격을 확인한 뒤 사용자의 포인트가 충분한지 확인합니다.
3. 시스템은 메뉴 가격을 차감하고 결제 주문을 하나 만들며 멱등성 기록을 하나의 일관된 업무
   작업으로 완료합니다.
4. 잔액이 부족하거나 트랜잭션이 실패하면 포인트를 차감하거나 결제 주문을 만들지 않습니다.
5. 완료된 같은 요청을 반복하면 포인트를 다시 차감하거나 주문을 다시 만들지 않고 최초 응답을
   반환합니다.
6. 결제 트랜잭션은 대기 수집 전송 상태의 주문을 만듭니다.
7. 커밋 뒤 시스템은 주문 ID, 사용자 식별자, 메뉴 ID, 결제 금액을 MockAPI.io의 `orders`
   리소스로 거의 실시간으로 전송합니다.
8. 각 전송 시도는 MockAPI.io를 호출하기 전에 아직 대기 상태인 주문만 잠급니다. 전송 실패는
   주문을 대기 상태로 남기고, 공유 스케줄러가 로컬 주문을 만들거나 포인트를 다시 차감하지 않은
   채 재시도합니다.
9. 클라이언트는 먼저 `orderId`로 MockAPI.io의 `orders` 리소스를 필터링합니다. payload가 같은
   레코드가 있으면 전송을 완료하고, 없으면 `POST /orders`로 레코드를 생성합니다.
10. 조회 결과가 여러 건이거나 같은 `orderId`의 주문 데이터가 다르면 외부 데이터 충돌로
    처리하고 로컬 전송 상태를 대기로 유지합니다.

### 시나리오 4: 인기 메뉴 조회

1. 사용자가 인기 메뉴 목록을 요청합니다.
2. 시스템은 현재 날짜 직전 완료 7일의 Redis 인기 메뉴 수를 합산합니다.
3. 시스템은 주문 수 내림차순으로 최대 3개 메뉴를 반환합니다.
4. Redis 인기 메뉴 캐시가 없거나 유효하지 않으면, 응답 전에 커밋된 MySQL 주문으로 캐시를
   재구축합니다.
5. 여러 인스턴스와 동시 주문 상황에서도 결과는 정확해야 합니다.

## 3. 기술 스택

의존성 버전과 라이브러리는 `build.gradle`을 기준으로 합니다.

| 영역 | 기술 | 용도 |
|---|---|---|
| 언어 | Java 21 | 애플리케이션 구현 |
| 빌드 | Gradle Wrapper | 재현 가능한 빌드와 테스트 실행 |
| 프레임워크 | Spring Boot 4.1.0 | 애플리케이션 설정과 런타임 |
| API | Spring Web MVC | HTTP API 구현 |
| 검증 | Spring Validation | 요청과 업무 입력 검증 |
| 영속성 | Spring Data JPA | 관계형 데이터 접근 |
| 주 데이터베이스 | MySQL 8.4 | 사용자, 메뉴, 주문, 멱등성 기록 |
| 캐시 | Spring Data Redis / Redis 7.4 Alpine | 일별 인기 메뉴 ZSET 캐시 |
| 외부 API | MockAPI.io | 결제 주문 수집 데이터를 저장하는 REST 리소스 |
| 운영 | Spring Boot Actuator | 헬스 및 운영 엔드포인트 |
| 개발 데이터베이스 | H2 | 경량 개발·테스트 지원 |
| 테스트 | JUnit Platform 및 Spring Boot 테스트 스타터 | 단위, 통합, 동시성, 제약 테스트 |
| 보일러플레이트 감소 | Lombok | 모델과 컴포넌트 보일러플레이트 감소 |
| 로컬 인프라 | Docker | 로컬 Redis 컨테이너 |

정확성은 인메모리 상태에 의존하면 안 됩니다. 공유 MySQL과 Redis 인프라가 여러 애플리케이션
인스턴스를 지원해야 합니다.

## 4. 도메인 모델

### 핵심 모델

| 모델 | 핵심 속성 | 책임과 제약 |
|---|---|---|
| 사용자 | `id`, `pointBalance` | 고객을 식별하고 현재 포인트 잔액을 보관하며 주문합니다. 잔액은 음수가 될 수 없습니다. |
| 메뉴 | `id`, `name`, `price` | 주문 가능한 커피 메뉴입니다. 가격은 양수여야 합니다. |
| 주문 | `id`, `userId`, `menuId`, `paymentAmount`, `orderedAt`, 전송 상태, 인기 메뉴 투영 상태 | 완료된 커피 구매를 나타내며 결제 시점의 가격과 수집 전송·Redis 투영 수명주기를 보존합니다. |
| 멱등성 기록 | `operation`, `idempotencyKey`, `requestHash`, `status`, `responseBody` | 여러 인스턴스에서 포인트 충전·결제 재시도가 중복 부작용을 만들지 않게 합니다. |
| 인기 메뉴 뷰 | 일별 ZSET 키, `menuId` 멤버, 결제 주문 수 점수 | 메뉴 순위를 위한 Redis 파생 캐시입니다. MySQL 주문이 기준 데이터입니다. |

### 인기 메뉴 뷰 저장 모델

인기 메뉴 뷰는 `Asia/Seoul` 업무 시간대의 달력 날짜마다 하나의 Redis ZSET으로 저장합니다.

| Redis 요소 | 형식 | 의미 |
|---|---|---|
| 키 | `popular-menu:{yyyy-MM-dd}` | 예: `popular-menu:2026-07-10`인 일별 인기 메뉴 버킷 |
| 타입 | ZSET | 숫자 점수로 정렬된 멤버 유지 |
| 멤버 | `menu:{menuId}` | 안정적인 메뉴 식별자. 이름과 가격은 MySQL에 보관 |
| 점수 | 결제 주문 횟수 | 해당 날짜에 성공적으로 결제된 메뉴 주문 수 |
| Projection marker | `popular-menu:projection:{orderId}` | 재시도 projection의 중복 증가 방지 |
| 기준 데이터 | MySQL의 `coffee_order` 행 | 누락되거나 유효하지 않은 캐시를 재구축하는 데이터 |

결제 커밋 후 애플리케이션은 주문의 projection marker 생성과 해당 일별 ZSET 점수 증가를
원자적으로 수행합니다. 주문은 Redis 투영이 확인될 때까지 대기 투영 상태를 보존하므로, 실패한
투영을 중복 집계 없이 재시도할 수 있습니다. 상위 3개 조회는 현재 날짜 직전 완료 7일 버킷을
합산하며 현재 날짜 ZSET은 포함하지 않습니다. 필요한 키가 없거나 사용할 수 없거나 유효하지
않으면 MySQL 주문을 집계해 캐시를 재구축합니다.

### 관계

- 사용자는 하나의 현재 포인트 잔액을 가집니다.
- 사용자는 여러 주문을 할 수 있습니다.
- 현재 API는 하나의 `menuId`를 받으므로 주문은 정확히 하나의 메뉴를 참조합니다.
- 메뉴는 여러 주문에서 참조될 수 있습니다.
- 인기 메뉴 뷰는 날짜와 메뉴별 MySQL 결제 주문에서 파생된 Redis 캐시입니다.
- 일별 ZSET에는 여러 메뉴가 있고, 같은 메뉴는 주문된 각 날짜 버킷에 나타날 수 있습니다.

### 업무 규칙

- 포인트만 지원되는 결제 수단입니다.
- 충전 금액과 메뉴 가격은 양수여야 합니다.
- 사용자 포인트 잔액은 음수가 될 수 없습니다.
- 주문의 결제 금액은 양수이고 결제 시점의 메뉴 가격을 보관합니다.
- 현재 API는 하나의 `menuId`를 받으므로 하나의 결제 주문은 하나의 메뉴 구매를 나타냅니다.
- 포인트 차감, 주문 생성, 멱등성 완료는 사용자가 보기에 원자적이어야 합니다.
- 성공적으로 결제된 주문만 영속화합니다.
- 동시 충전과 결제에서 갱신 유실이나 초과 결제가 발생하면 안 됩니다.
- 같은 클라이언트 멱등키와 요청의 충전·결제 재시도는 다른 부작용 없이 최초 응답을 반환해야
  합니다.
- 같은 키를 다른 요청에 재사용하면 거부합니다.
- 주문은 완료된 결제 트랜잭션에서 영속적인 대기 수집 전송 상태로 생성됩니다. 첫 전송은 커밋
  뒤에 시도하며 전송 실패는 결제를 롤백하지 않고 나중에 재시도합니다.
- 각 전송 시도는 외부 호출 전에 새 DB 트랜잭션에서 아직 대기 상태인 주문을 잠급니다. 이로써
  정상적인 인스턴스 경합에서 같은 주문을 동시에 전송하지 않습니다.
- 외부 수집 payload에는 조정 식별자인 주문 ID와 사용자 ID, 메뉴 ID, 결제 금액을 포함합니다.
- MockAPI.io 레코드를 생성하기 전에 클라이언트가 `GET /orders?orderId={orderId}`를 호출합니다.
  주문 데이터가 같은 레코드 한 건이 있으면 이미 완료된 전송으로 처리하고, 결과가 비어 있으면
  `POST /orders`를 호출합니다.
- 수집 전송 실패는 나중에 성공할 때까지 대기 상태로 남습니다. 외부 호출과 로컬 `SUCCEEDED`
  커밋은 원자적이지 않으므로 재시도마다 생성 전에 조회합니다. MockAPI.io는 로컬 DB 트랜잭션이나
  `orderId` 유일성을 보장하지 않으며, 수동 또는 무관한 외부 쓰기는 애플리케이션의 일관성 경계
  밖에 있습니다.
- 인기 메뉴에는 `Asia/Seoul` 기준 현재 날짜 직전 완료 7일의 성공 결제 주문만 사용합니다.
- 커밋된 주문은 일별 Redis ZSET 점수를 정확히 한 번 증가시킵니다.
- MySQL 결제 주문이 기준 데이터이며, 캐시가 없거나 사용할 수 없거나 유효하지 않으면 Redis
  일별 ZSET을 재구축합니다.
- 인기 메뉴 수는 모든 실행 인스턴스에서 일관돼야 합니다.
- 주문 수가 같으면 메뉴 ID 오름차순으로 정렬합니다.

## 5. 기능 목록

### 필수 기능

| ID | 기능 | 요구사항 | 인수 기준 |
|---|---|---|---|
| F-01 | 커피 메뉴 목록 | 메뉴 ID, 이름, 가격을 제공합니다. | 요청이 필요한 필드를 가진 모든 메뉴를 반환합니다. |
| F-02 | 포인트 충전 | 사용자 식별자와 충전 금액을 받고 `1원 = 1포인트`를 적용하며 멱등키를 요구합니다. | 유효한 충전은 올바른 사용자 잔액을 정확히 한 번 증가시킵니다. 같은 키·요청 재실행은 최초 결과를 반환합니다. |
| F-03 | 커피 주문·결제 | 하나의 요청에서 사용자 식별자와 메뉴 ID를 받고 멱등키를 요구하며 포인트로만 결제합니다. | 성공 요청은 현재 메뉴 가격을 원자적으로 차감하고 주문 하나와 멱등성 결과를 저장합니다. 재시도는 중복 주문·차감을 만들지 않습니다. |
| F-04 | 실시간 주문 데이터 전송 | 성공 주문마다 대기 전송 상태를 저장하고, 커밋 뒤 주문 ID·사용자 ID·메뉴 ID·결제 금액을 MockAPI.io와 조정합니다. | 전송 시도는 아직 대기 상태인 주문을 잠그고, MockAPI.io에서 `orderId`로 기존 레코드를 찾은 뒤 일치 항목이 없을 때만 생성합니다. 실패는 영속 주문 데이터를 롤백하지 않습니다. |
| F-05 | 인기 메뉴 목록 | 직전 완료 7개 달력 날짜의 주문 수가 많은 메뉴 3개를 반환합니다. | 현재 날짜를 제외하고 Redis 일별 캐시 수를 합산하며, 누락되거나 유효하지 않은 캐시는 MySQL에서 재구축해 반환 수를 바꾸지 않습니다. |
